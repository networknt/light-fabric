§00  执行摘要
  • 定位：基于 tokio / axum / Cloudflare Pingora 的企业 AI 治理平台，配套外部项目 light-portal 作为统一控制面；全仓库 114k 行 Rust、707 个测试函数、43 个 workspace 成员。

§01

项目概览

官方文档将 light-fabric 定义为「高性能的企业 AI 服务统一平台」，管理从 agent、tool、skill、memory 到 MCP server、API、gateway、workflow 的完整生命周期。

设计哲学围绕四条原则：统一控制面（light-portal 作为发现、治理、审计的单一事实源）、企业治理优先（安全与策略执行优于去中心化自治）、集成生态（分布式组件织成可观测的整体——「fabric」之名的由来）、持久身份（基础设施层面的长期身份体系）。技术栈为 Rust edition 2024，异步基座 tokio，服务框架 axum，网关数据面构建在 Cloudflare Pingora 之上，TLS 用 rustls，持久化用 sqlx / Postgres（含 pgvector）。

§04

关键子系统

MCP 路由：一个端点、两种 profile

网关在同一个 /mcp 端点同时承载两种协议形态，按请求特征分流，并提供 reject-by-default 的 stateless↔legacy 桥接与条件化自动发现：

Legacy stateful（默认，生产级）Stateless 2026-07-28（RC）
会话initialize + Mcp-Session-Id，后端会话映射与清理无会话、无 initialize，每请求自描述
协商握手协商 2024-11-05 / 2025-03-26 / 2025-06-18 / 2025-11-25MCP-Protocol-Version / Mcp-Method / Mcp-Param-* 请求头
方法面tools/list · tools/call（授权感知的 tools-list 缓存）server/discover · tools/list · tools/call，resultType / ttlMs / cacheScope
校验—JSON Schema 2020-12 出入参校验、x-mcp-header 注解、origin 白名单、主体指纹绑定
状态生产级RC · f9d8716 刚启用

当前方法面是 tools-only：resources / prompts / sampling / elicitation / logging 均未实现（源码中仅出现在拒绝路径），subscriptions/listen（SSE 流式推送）已有响应体脚手架但被明确 gate。这是 §06 中 P1 级开发空间的主体。

安全体系

  • 认证：JWT（JWK 缓存、audience / issuer / scope、过期模式）、出站 OAuth 四流程、MSAL / Entra ID + BFF token 交换、SPA Cookie 会话（CSRF）、API Key、Basic、无状态认证、按路径前缀的统一安全策略。
  • 授权与数据：FGA 细粒度授权、CEL 规则引擎、MCP tools-list 可见性过滤、响应行 / 列过滤、PII 令牌化、受保护路径策略、SSRF 目的地策略、委托授权令牌（agent-delegation）。
  • 已知缺口：SWT 不透明令牌 introspection 返回 501；出站 client 的 mTLS / sign / deref / 熔断重试在设计文档中标注「Not yet modeled」（详见 §06）。

model-provider：统一 LLM 抽象

单一 Provider trait 覆盖一次性 / 多轮对话、原生工具调用、视觉、prompt 缓存与能力探测；reliable 元 provider 提供重试 / 退避 / 故障转移，router 元 provider 按复杂度路由。工具调用支持现状：

Provider原生工具调用说明
openai · anthropic · azure_openai · ollama · openrouter · copilot · compatible（含 telnyx）支持完整实现，作为补全其余 provider 的参照
gemini · bedrock · glm运行时报错声明 native_tool_calling: false，调用即 bail!("... not yet implemented")
claude_code · codex 等 CLI 适配器未启用CLI 桥接形态，能力声明为关闭

沙箱执行子系统

平台最大的单项投入（约 80 个 commit）。ExecutionBackend trait 定义完整生命周期——prepare / execute / inspect / cancel / collect_artifacts / read_logs / checkpoint / restore / cleanup / reconcile_orphans——错误分类中显式包含 Unknown 以支持 at-least-once 对账。三个后端（Cube microVM 真实后端、rootless OCI、mock）共享同一 conformance 套件；每周 CI 在自托管 runner 上跑真实 Cube / Docker 矩阵。安全侧由 GrantLedger（尝试级授权台账）、SSRF 目的地策略、TrustBundle 与摘要绑定的 immutable-package 兜底；高价值动作（发布 / 签名 / 建分支 / PR）走独立的 fixed-action 通道，支持 sha1 + sha256 双 Git 对象格式。

工作流与运行器

workflow-core 实现 Serverless Workflow DSL 模型（任务 / 事件 / 重试 / 超时 / schema / 目录）。light-workflow 编排器包含事务化审批、artifact 发布与保留策略、provenance 记录、runner 调度、结果 / 会话对账器与租约回收器，Postgres 持久化并有集成测试。light-workflow-runner 是租户侧代理：supervisor 管理 worker 进程、broker 中转、journal 落盘、staging 物料化，经 WebSocket 与编排器通信，提供 Cube / Docker / rootless-OCI / mock 四种示例配置。

控制面接入

portal-registry 通过 WebSocket 上的 JSON-RPC 完成注册、发现与缓存控制，失败时回退直连注册表；rkyv 二进制编码作为实验通道（wire 格式硬性锁定 little-endian + 32 位指针宽度，由 scripts/check-controller-wire-*.sh 拒绝不支持的构建配置）。controller-wire 是冻结的传输无关编解码 crate——零传输 / 异步依赖，自带 conformance 与安全模糊测试。配置层支持远程配置中心引导 + 运行期 reload，每个网关 handler 注册独立的热重载器。

P0Hindsight 记忆：让头条功能名副其实

Embedding 服务 + 真实语义召回

README 头条能力「biomimetic memory banks」目前的召回路径是：以 384 维零向量调用 recall()，而 SQL 按 embedding <=> $3（pgvector 余弦距离）排序——查询向量为零时排序完全无意义。源码注释自述「we don't have an embedding service yet」。工作项：为 model-provider 增加 embedding 能力（trait 扩展或独立 provider），接入 recall / retain 双路径，并为 hindsight-client（当前 0 测试）补 Postgres + pgvector 集成测试。

  • apps/light-agent/src/main.rs:3889-3894（zero-vector 注释与调用）
